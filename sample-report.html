1 · Scope & methodology

This review covered the full application firmware for a battery-powered connected sensor: sensor acquisition, FreeRTOS task structure, the BLE/Wi-Fi connectivity layer, OTA update path, and the power-management state machine. The bootloader and vendor SDK were treated as out of scope except where the application interacts with them.

Method: manual line-by-line review of safety- and reliability-relevant paths, supported by static analysis (cppcheck + clang-tidy with a MISRA-oriented profile), compiler-warning audit at -Wall -Wextra, and a targeted concurrency review of all data shared between ISRs, tasks, and DMA. No hardware was instrumented for this engagement.

2 · Executive summary

The firmware is well-structured and the team clearly understands the product domain. However, the review found two critical defects that can cause field hangs or silent data corruption, plus several high/medium issues that undermine long-term reliability. None require a redesign — all are addressable within the remediation plan in §5. Overall risk posture: Elevated — not yet field-ready until C-1 and C-2 are resolved.

3 · Critical findings (detail)

Critical Unprotected ISR/task shared buffer C-1 · sensor_isr.c

ConcurrencyMISRA C:2012 Dir 4.1Data race

Issue

The sample ring buffer is written by ADC_IRQHandler() and read by vAcquisitionTask(). The index variables are plain uint16_t with no volatile qualifier and no critical section. On a 32-bit core a partially updated index can be read, and the compiler may cache the value in a register across the loop — yielding stale or torn reads, intermittently and unreproducibly in the lab.

Impact

Silent corruption of sensor data and occasional missed samples. The failure rate scales with acquisition rate, so it tends to appear in the field, not on the bench — the worst kind of bug.

Evidence
// sensor_isr.c
static uint16_t head, tail;        // not volatile; shared ISR <-> task
void ADC_IRQHandler(void){ buf[head++] = ADC1->DR; }     // no wrap guard, no barrier
Recommendation

Qualify shared state as volatile, make the index update atomic (single-writer ISR, single-reader task is fine if reads/writes are atomic and bounded), and wrap the multi-field update in a short critical section (taskENTER_CRITICAL() / __disable_irq()). Add the missing ring-buffer wrap. Prefer a lock-free SPSC queue pattern documented for the team.

Critical Unbounded copy of network payload C-2 · net_rx.c

CWE-120 Buffer OverflowMISRA C:2012 Rule 21.17Security

Issue

Incoming frame length is taken from the wire and used to copy into a 256-byte stack buffer with no check that it fits. A malformed or malicious packet overflows the stack frame.

Impact

Remote stack smashing — at minimum a crash/DoS, at worst code execution on a connected device. For an internet-reachable product this is a shipping blocker.

Evidence
// net_rx.c
uint8_t frame[256];
memcpy(frame, pkt->data, pkt->len);   // pkt->len attacker-controlled, no bound check
Recommendation

Validate pkt->len <= sizeof(frame) before the copy and reject oversized frames; replace with a length-checked copy helper used project-wide. Treat all wire-sourced lengths as untrusted at the boundary, and add a fuzz/negative test for the parser.

4 · High findings (detail)

High Watchdog disabled across OTA H-1 · ota.c

ReliabilityIEC 61508 — safe state

Issue & impact

The independent watchdog is stopped for the full OTA flow "to avoid spurious resets during flash writes." If the transfer stalls (lost link, power sag), the device hangs with no recovery path until a human power-cycles it — unacceptable for a remote/connected product.

Recommendation

Keep the watchdog running; refresh it at bounded checkpoints between flash operations. Add a transfer timeout that rolls back to the known-good image, and verify the new image (CRC/signature) before marking it bootable. Power-loss during OTA should always leave a bootable device.

High Blocking wait inside ISR H-2 · i2c_drv.c

Real-timeMISRA C:2012 Dir 4.1

Issue & impact

A busy-wait on an I²C status flag runs inside interrupt context. If the bus is held (stuck slave, missing pull-up), the ISR spins, blocking all lower-priority interrupts and the scheduler — a system-wide stall from a single peripheral fault.

Recommendation

Move the transfer to a non-blocking interrupt/DMA-driven state machine with a timeout and bus-recovery sequence. ISRs should do minimal, bounded work and hand off to a task.

High Fault handler does nothing H-3 · main.c

Fault handlingDiagnostics

Issue & impact

HardFault_Handler() is while(1){}. On a fault the device freezes with no diagnostics and no defined safe state — field failures are uninvestigable and the product doesn't fail safe.

Recommendation

Capture the stacked fault context (PC, LR, PSR, fault status registers) to a persistent region, drive outputs to a defined safe state, then perform a controlled reset. Surface the fault record in the next session for field post-mortem.
